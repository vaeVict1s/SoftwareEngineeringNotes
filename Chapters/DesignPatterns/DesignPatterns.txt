*** *** *** CREATIONAL PATTERNS *** *** ***
-) FACTORY METHOD
Interface for creating objects in a superclass, 
    keeping subclasses free to alter the type of objects that will be created
Achieved replacing the direct object construction calls 
    with calls to a special Factory Method.
It is possible therefore to override the subclasses' Factory Method 
    in order to change the class of objects being created by the method itself.
The objects produced by different classes might be different 
    only if the latter have a common base class or interface.
The Factory Method in the base class should have 
    its return type declared as part of the interface 

-) ABSTRACT FACTORY
Makes easier to produce families of related items
    without specifying their concrete classes.
Achieved declaring explicit interfaces for each distinct product of the product family
    All variants of products need to follow those interfaces.
An Abstract Factory is finally declared with 
    a list of creation methods for all the items that are part of the product family 
For each variant of a product family, 
    a separate factory class based on the AbstractFactory interface will be needed.
Client accesses with both factories and products via their abstract interfaces.

-) BUILDER
Facilitates the construction of complex objects step by step,
    producing different types and representations of an object
    using the same construyction code.
Achieved extracting the construction code out of its own class
    and moving it to a separate Builder classe.
Construction is managed with a set of steps executed in series 
    and only in case they are necessary for producing a specific configuration of the object.
A specific Builder class could be implemented to manage
    the same building steps in a different manner
A Director class, not mandatory, can be implemented to collect the steps in a specific order,
    while the Builder provides the implementation of what to do.

-) PROTORYPE
Enable the copy of existing objects without implementing code dependent on their classes.
Achieved delegating the cloning process to the actual objects that are being cloned.
    A common interface is implemented for all the objects that support cloning.
    The clone method is similar for all classes, 
    it creates an object of the current class and carries over all the fields values of the old cloned one.
    Copying private fields is possible for any programming language which allows to access private fields of other objects of the same class.
An object which could be cloned is called Prototype.
Cloning is an alternative to subclassing for objects with many fields and configurations.

-) SINGLETON
Ensure that a class could have one instance only, 
    while providing a global access point to this instance.
    This functionality could be interpreted as a violation of the Single Responsibility Principle 
The implementation of the Singleton has 2 steps in common:
    -) default constructor is made private, to prevent other objects from calling it
    -) a static creation method acts as a constructor, calling the private constructor
       to create an object and save it in a static field.
       Following calls to this method will return the cached object. 


*** *** *** STRUCTURAL PATTERNS *** *** ***
-) ADAPTER
Allows the collaboration among objects with incompatible interfaces.
Achieved through the implementation of an Adapter which make 
    one interface understandable by an object of a different class.
    The adatper gets an interface compatible with one existing object.
    Through this interface, the objects can call the adapter's methods.
    Upon receiving a call, the adapter passes the request to the second object,
    in a format and order it could understand.
    An Adapter could work in both directions.

-) BRIDGE
This pattern splits a large class or a set of closely related classes into two separate hierarchies which can be developed independently of each other.
Achieved by switching from inheritance to object composition.
    One of the dimensions is extracted into a separate class hierarchy, 
    therefore the original classes will reference an object of the new hierarchy
    instead of having all its states and behaviors coded within the class.
    The original class implement a reference field for each "extracted-dimension" classes 
    This extraction makes a class able to delegate any 'dimension-related' work
    to the linked 'dimension-related' object.

-) COMPOSITE
After enabling composition of objects into tree structures,
this pattern let clients work with these structures as they were individual objects.
Achieved extending the class

-) DECORATOR
Attaches new behaviors to objects by placing these objects inside
special wrapper objects containing the desired behavior.

-) FACADE
A simplified interface to a complex subsystem which contains lots of moving parts.
Functionalities provided by the Facade could be reduced to the subset the client is interested in.

-) FLYWEIGHT
Reduces RAM consumption by sharing common parts of state between multiple objects,
instead of creating copies of same data in each object.

-) PROXY
Provides with a substitute or placeholder for another object,
managing the access to the orignal object and making the client able
to perform actions either befor or after the request gets through to the original object.
Achieved implementing a new proxy class with the same interface as an original service object. Clients of the original objects could passed the proxy object, which manage the real service object and delegates work.

*** *** *** BEHAVIORAL PATTERNS *** *** ***

-) CHAIN OF RESPONSIBILITY
Makes possible to pass a request along a chain of handlers,
each of which decides upon receiving a request either to process the request
or to pass it to the next handler in the chain

-) COMMAND


-) ITERATOR
Enables the client to traverse elements of a collection without exposing its underlying representation.
Achieved extracting the traversal behavior of a collection into a separate object, called iterator.
Details of the trasversals are encapsulated into the iterator class;
in this way several iterators can go through the same collection at the same time
and independently of each other.
Iterators should implement the same interface in order to make the client code compatible with any collection type or any trasversal algorithm as long as there's a proper iterator. 

-) MEDIATOR
Reduces chaotic and confusing dependencies between objects, restricting direct communications between the objects and forcing them to collaborate only through a mediator object.
Achieved cutting all direct communication between the components which are required to be independent of each other nd
