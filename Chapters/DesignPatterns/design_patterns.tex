\section{Design Patterns}
\label{sec:designpatterns}
\vspace{10.5cm}
\textbf{Design Patterns} describe how to solve recurring design problems 
to design flexible and reusable object-oriented software, that is, 
objects that are easier to implement, change, test, and reuse.

\subsection{Creational Patterns}

    \subsubsection{Factory Method}
    This pattern uses \textbf{factory methods} to deal with the problem of 
    \textbf{creating objects without having to specify the exact class} of 
    the object that will be created.
    \newline
    Objects creation is achieved by calling a factory method --- either specified 
    in an interface and implemented by child classes, or implemented 
    in a base class and optionally overridden by derived classes --- 
    rather than by calling a constructor.
    
    \nparagraph{When to use this pattern} 
    The object's creation may lead to a significant duplication of code, 
    may require information not accessible to the composing object, 
    may not provide a sufficient level of abstraction, 
    or may otherwise not be part of the composing object's concerns.\\
    The factory method design pattern handles these problems by defining 
    a \textbf{separate method for creating the objects}, which subclasses 
    can then override to specify the derived type of product 
    that will be created.\\
    Following a list of use cases
    \begin{itemize}
        \item the exact types and dependencies of the objects your code should work with are not known
        \item there is the need to provide users of a library or framework with a way to extend its internal components
        \item there is the concern to save system resources by reusing existing objects instead of rebuilding them each time.
    \end{itemize}
    
    \nparagraph{Structure}
    \begin{figure}[H]
        \includegraphics[scale=0.5]{CreationalPatterns/Factory_Method_Pattern_Class_Diagram}
        %~ \missingfigure[figwidth=0.5\textwidth]{Missing Class Diagram}
        \label{fig:cd_factory-method}
        \caption{Factory Method Class Diagram}
    \end{figure}
    The structure of the Factory Method is quite easy
    \begin{itemize}
        \item \textbf{Product} defines the interface common to all the objects.
        \item{} \textbf{Concrete Products} are the different implementations of the previous interface.
        \item{} \textbf{Creator} is the base class which declares the factory method.
        \begin{itemize}
            \item \textbf{factory method} should either be abstract or return a default product.            
            \item \textbf{returned type} must match the interface
            \item 
        \end{itemize}
        \item \textbf{Concrete Creators} override the base factory method 
        in order to return different types of \textbf{Concrete Products}, 
        which could not be a new instance every time Factory methods could also return 
        existing objects from a cache, an object in a pool or similar ones.
    \end{itemize}
    
    Factory Method helps decouple the creation of the different concrete objects 
    from the core business logic the Creator might and usually does have.

    \subsubsection{Abstract Factory}
    This pattern makes it easier to produce families of related items
    without specifying their concrete classes.\\
    This is achieved declaring \textbf{explicit interfaces for each distinct product} of the product family and making 
    \textbf{all variants of products need to follow those interfaces}.\\
    An Abstract Factory is finally declared with a \textbf{list of creation methods} 
    for all the items that are part of the product family. For each variant of a product family, 
    \textbf{a separate factory class} based on the AbstractFactory interface will be needed 
    and \textbf{clients access with both factories and products via their abstract interfaces}.
    
    \nparagraph{When to use this pattern}
    \begin{itemize}
        \item
        \item
        \item
    \end{itemize}
    
    \nparagraph{Structure}
    \begin{figure}[H]
        \includegraphics[scale=0.3]{CreationalPatterns/Abstract_Factory_Pattern_Class_Diagram}
        %~ \missingfigure[figwidth=0.5\textwidth]{Missing Class Diagram}        
        \label{fig:cd_abstract_factory}
        \caption{Abstract Factory Class Diagram}
    \end{figure}
    \begin{itemize}
        \item \textbf{Abstract Products} declare the interfaces for a set or family 
              of related but distinct products.
        \item \textbf{Concrete Products} are the implementations of the concrete variants 
              of abstract products. Each single abstract product must be implemented in all given variants.
        \item \textbf{Abstract Factory} is the interface to collect 
              the declarations of sets of methods to create each of the abstract products.
        \item \textbf{Concrete Factorys} implement the creation methods of the 
              abstract factory, each creating only the product variant it correspons to. 
        \item \textbf{Client} can work with any concrete factory as long as it communicates with 
              the objects through their abstract interfaces. In order to have this behavior, 
              the signatures of the concrete factories' creation methods must match the 
              corresponding abstract interfaces'. 
    \end{itemize}

    \subsubsection{Builder}
    \textbf{Builder} facilitates the \textbf{construction of complex objects step by step},
    producing different types and representations of an object using the same construyction code.
    This is achieved extracting the construction code out of its own class
    and moving it to a separate Builder class. Construction is managed 
    with a set of steps executed in series and only in case they are necessary 
    for producing a specific configuration of the object. A specific Builder class 
    could be implemented to manage the same building steps in a different manner. 
    A Director class, not mandatory, can be implemented to collect 
    the steps in a specific order, while the Builder provides 
    the implementation of what to do.
    
    \nparagraph{When to use this pattern}
    \begin{itemize}
        \item
        \item
        \item
    \end{itemize}

    \subsubsection{Prototype}
    \textbf{Prototype} enables the copy of existing objects without implementing 
    code which depends on their classes. This is achieved delegating the cloning process 
    to the actual objects that are being cloned. A common interface is implemented 
    for all the objects that support cloning. The clone method is similar for all classes: 
    it creates an object of the current class and carries over all the fields values of the old cloned one.
    Copying private fields is possible for any programming language which allows to access private fields of other objects of the same class.
    An object which could be cloned is called Prototype. 
    Cloning is an alternative to subclassing for objects with many fields and configurations.

    \nparagraph{When to use this pattern}

    \subsubsection{Chain of Responsibility}
    \textbf{Chain of Responsibility} makes possible to pass a request 
    along a chain of handlers, each of which decides upon receiving 
    a request either to process the request or to pass it to the next handler in the chain.
    
    \nparagraph{When to use this pattern}
    \begin{itemize}
        \item
        \item
        \item
    \end{itemize}
    
    \nparagraph{Structure}
    \begin{figure}[H]
        \includegraphics[scale=0.5]{BehavioralPatterns/Chain_of_Responsibility_Pattern_Class_Diagram}
        %~ \missingfigure[figwidth=0.5	\textwidth]{Missing Class Diagram}        
        \label{fig:cd_chain_of_responsibility}
        \caption{Chain of Responsibility Class Diagram}
    \end{figure}
    \begin{itemize}
        \item \textbf{Handler} declares an common interface for all the concrete handlers, 
        usually in terms of a first method for handling the requests and second one 
        to set the next handler in the chain.  
        \item \textbf{Base Handler} could be optionally implemented 
        \item \textbf{Concrete Handlers} contain the actual code to process the request, 
        which will be executed after an optional preliminary check 
        to understand whether it could be processed or passed on.
        \item \textbf{Client} may create chains both statically and dinamically, and 
        depending on its own logic as well. The request can be sent to any handler in the chain.
    \end{itemize}

    \subsubsection{Iterator}
    \textbf{Iterator} enables the client to traverse elements of a collection 
    without exposing its underlying representation. This is achieved 
    extracting the traversal behavior of a collection into a separate object, called iterator.
    Details of the trasversals are encapsulated into the iterator class; 
    in this way several iterators can go through the same collection 
    at the same time and independently of each other.
    Iterators should implement the same interface in order to make the client code 
    compatible with any collection type or any trasversal algorithm, 
    as long as there's a proper iterator. 

    \nparagraph{When to use this pattern}
    \begin{itemize}
        \item
        \item
        \item
    \end{itemize}
    
    \nparagraph{Structure}
    \begin{figure}[H]
        \includegraphics[scale=0.5]{BehavioralPatterns/Iterator_Pattern_Class_Diagram}
        %~ \missingfigure[figwidth=0.5	\textwidth]{Missing Class Diagram}        
        \label{fig:cd_iterator}
        \caption{Iterator Class Diagram}
    \end{figure}
    \begin{itemize}
        \item \textbf{Iterator} interface declares the operations needed 
        to traverse a collection, like fetching the next element, keeping track of the current position 
        and so on.
        \item \textbf{Concrete Iterator} implements specific algorithms 
        each undependently on others in order to be able to use many of them 
        at the same time. 
        \item \textbf{Collection} interface declares one or multiple methods 
        for getting iterator compatible with the collection. The return type 
        of these methods must match the Iterator interface.
        \item \textbf{Concrete Collection} returns instances of 
        a particular concrete iterator class on client's request.
        \item \textbf{Client} can work with both the collection and the iterator 
        using their interfaces.
    \end{itemize}

    \subsubsection{Mediator}
    \textbf{Mediator} reduces chaotic and confusing dependencies between objects, 
    restricting direct communications between the objects and forcing them 
    to collaborate only through a mediator object. This is achieved cutting 
    all direct communication between the components which are required 
    to be independent of each other abd ...

    \nparagraph{When to use this pattern}
    \begin{itemize}
        \item
        \item
        \item
    \end{itemize}
    
    \nparagraph{Structure}
    \begin{figure}[H]
        \includegraphics[scale=0.5]{BehavioralPatterns/Mediator_Pattern_Class_Diagram}
        %~ \missingfigure[figwidth=0.5	\textwidth]{Missing Class Diagram}        
        \label{fig:cd_mediator}
        \caption{Mediator Class Diagram}
    \end{figure}
    \begin{itemize}
        \item \textbf{Components} are classes containing some business logic. 
        Each one has a reference to a mendiator, whose referenced type is 
        the Mediator interface's. Components are unaware of the actual Meditor class, 
        so that it could be reused in other contexts with different mediators.
        \item \textbf{Mediator} interface declares the communication structure 
        between components. Components may pass any context as argument of their methods, 
        although they must avoid coupling the receiving component with the sender.
        \item \textbf{Concrete Mediator} implement relation between various components. 
        It could be useful to keep references to all the components managed 
        and their lifecycle as well. 
    \end{itemize}
    Components are not aware of each other. In case some communication needs 
    to go from a Component to a different one, it must pass through the Mediator. 
    Likewise, the sender doesn't know who will handle the request as well as the 
    receiver doesn't know who sent the request.

    \subsubsection{Adapter}
    \textbf{Adapter} allows the collaboration among objects with incompatible interfaces.
    This is achieved through the implementation of an Adapter which make 
    one interface understandable by an object of a different class.
    The adatper gets an interface compatible with one existing object.
    Through this interface, the objects can call the adapter's methods.
    Upon receiving a call, the adapter passes the request to the second object,
    in a format and order it could understand.
    An Adapter could work in both directions.

    \nparagraph{When to use this pattern}
    \begin{itemize}
        \item
        \item
        \item
    \end{itemize}
    
    \nparagraph{Structure}
    \begin{figure}[H]
        \includegraphics[scale=0.5]{StructuralPatterns/Adapter_Pattern_Class_Diagram}
        %~ \missingfigure[figwidth=0.5	\textwidth]{Missing Class Diagram}        
        \label{fig:cd_adapter}
        \caption{Adapter Class Diagram}
    \end{figure}
    \begin{itemize}
        \item \textbf{Client} is the business logic of the program.
        \item \textbf{Client Interface} defines the protocol for a class to 
        be able to collaborate with it.
        \item \textbf{Service} is usually a 3rd party or legacy software 
        with an interface incopatible with the Client.
        \item \textbf{Adapter} is a class able to work with both the client and 
        the service. It receives calls from the client and translates 
        into calls to a wrapped service object.
    \end{itemize}
    Since the Client interacts with the adapter through the client interface, 
    it doesn't get coupled. Therefore, different adapters can be added to the program 
    without breaking any existing code.

    \subsubsection{Bridge}
    \textbf{Bridge} splits a large class or a set of closely related classes 
    into two separate hierarchies which can be developed independently of each other.
    This is achieved by switching from inheritance to object composition.
    One of the dimensions is extracted into a separate class hierarchy, 
    therefore the original classes will reference an object of the new hierarchy
    instead of having all its states and behaviors coded within the class.
    The original class implement a reference field for each "extracted-dimension" classes 
    This extraction makes a class able to delegate any 'dimension-related' work
    to the linked 'dimension-related' object.

    \nparagraph{When to use this pattern}
    \begin{itemize}
        \item
        \item
        \item
    \end{itemize}

        complex behavior that relies on primitive operations declared 
        by the implementation.
        \item \textbf{Concrete Implementations1} implement specific code.
        \item \textbf{Refined Abstraction} are optional and might help

        \item \textbf{Component} is the interface common for both wrapper and

        \item \textbf{Concrete Component} is a class being wrapped and whose 
        basic behavior can be altered by decorators.
        \item \textbf{Base Decorator} has a reference of the wrapped object,

        delegates all operations to the wrapped object.

        parent method.

    \textbf{Flyweight} reduces RAM consumption by sharing common parts of state 
    between multiple objects, instead of creating copies of same data in each object.

    \nparagraph{When to use this pattern}
    \begin{itemize}
        \item
        \item
        \item
    \end{itemize}
    
    \nparagraph{Structure}
    \begin{figure}[H]
        \includegraphics[scale=0.5]{StructuralPatterns/Flyweight_Pattern_Class_Diagram}
        %~ \missingfigure[figwidth=0.5	\textwidth]{Missing Class Diagram}        
        \label{fig:cd_flyweight}
        \caption{Flyweight Class Diagram}
    \end{figure}
    \begin{itemize}
        \item \textbf{Flyweight} contains the portion of the original object's state 
        that can be shared between multiple objects. The state stored 
        inside a flyweight is called \textbf{\textit{intrinsic}}; the state 
        passed to the flyweight's methods is called \textbf{\textit{extrinsic}}. 
        The same flyweight object can be used in many contexts.
        \item \textbf{Context} contains the extrinsic state, unique across all original 
        objects. It represents the full state when paired with one of the 
        flyweight objects.
        \item \textbf{Flyweight Factory} manages a pool of existing flyweights 
        freeing the Client from the task of creating. The Flyweight Factory 
        receives a call to retrieve a specific flyweight, looks over previously 
        created flyweight and either returns one of these or creates 
        a ne wone in case nothing matches the search criteria.
        \item \textbf{Client} sees the flyweight as a template object 
        which can be configured at runtime by passing some contextual 
        data into parameters of its methods.
    \end{itemize}
