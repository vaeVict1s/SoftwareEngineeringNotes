\chapter{Containers}
\label{chap:containers}
See~\cite{cppreference-containerslib}.
\newline
The Containers library is a \highlightit{generic collection of class templates and algorithms} 
that allow programmers \highlightit{to easily implement 
common data structures} like queues, lists and stacks.
\newline
The container classes are:
\begin{itemize}
    \item \highlightit{sequence containers} ---
          data structures that can be accessed sequentially.
    \item \highlightit{associative containers} ---
          searchable in $O(\log n)$ (typically tree-based, ordered by key).
    \item \highlightit{unordered associative containers} ---
          searchable in $O(1)$ on average and $O(n)$ in the worst case (hash-based).
    \item \highlightit{container adaptors} ---
          provide a restricted interface on top of an underlying container 
          (\texttt{stack}, \texttt{queue}, \texttt{priority\char`_queue}, and the C++23 \texttt{flat\char`_*} family).
\end{itemize}

\toclessnparagraph{When to use which}
\begin{itemize}
    \item Default to \texttt{std::vector} --- contiguous storage, best cache behaviour, random access $O(1)$.
    \item Use \texttt{std::array} for a fixed size known at compile time (no heap).
    \item Use \texttt{std::deque} when you need efficient push/pop at \textbf{both} ends 
          (not truly one contiguous block).
    \item Use \texttt{std::list} / \texttt{forward\char`_list} when you need stable iterators 
          under insert/erase in the middle, or splicing --- at the cost of locality.
\end{itemize}

\toclessnparagraph{\texttt{vector} capacity}
Growth may reallocate: all iterators/references are invalidated when capacity changes.
Prefer \texttt{reserve} when the final size is known (fewer allocations; ties to 
Chapter~\ref{chap:memmag} and Section~\ref{sec:avoiding-allocations}).

\begin{lstlisting}[language=C++]
std::vector<int> v;
v.reserve(1024);          // capacity >= 1024, size still 0
for (int x : data)
    v.push_back(x);       // no reallocation if data.size() <= 1024

v.shrink_to_fit();        // non-binding request to release unused capacity
\end{lstlisting}

    \hline
    \end{tabularx}
\end{table}

\toclessnparagraph{When to use which}
\begin{itemize}
    \item Need ordered iteration or predecessor/successor by key $\rightarrow$ \texttt{map}/\texttt{set}.
    \item Multiple equivalent keys $\rightarrow$ \texttt{multi*}.
    \item Lookup/insert typically $O(\log n)$; iterators stay valid except for erased elements.
\end{itemize}

\begin{lstlisting}[language=C++]
std::map<std::string, int> ages{{"Ann", 30}, {"Bob", 25}};
ages["Ann"] = 31;                 // insert or assign
if (auto it = ages.find("Bob"); it != ages.end())
    ++it->second;
\end{lstlisting}

    \begin{tabularx}{\textwidth}{|l|X|}

    \href{https://en.cppreference.com/w/cpp/container/unordered_set}{unordered\char`_set} & collection of \highlightit{unique keys, hashed by keys} \\        
    \hline
    \href{https://en.cppreference.com/w/cpp/container/unordered_map}{unordered\char`_map} & collection of \highlightit{key-value pairs, hashed by keys, keys are unique} \\        
    \hline
    \href{https://en.cppreference.com/w/cpp/container/unordered_multiset}{unordered\char`_multiset} & collection of \highlightit{keys, hashed by keys} \\        
    \hline
    \href{https://en.cppreference.com/w/cpp/container/unordered_multimap}{unordered\char`_multimap} & collection of \highlightit{key-value pairs, hashed by keys} \\        
    \hline
    \end{tabularx}
\end{table}

\toclessnparagraph{When to use which}
\begin{itemize}
    \item Prefer when you care about average $O(1)$ lookup and do \textbf{not} need order.
    \item Provide a good hash and equality for custom keys; watch load factor / rehash 
          (rehash invalidates iterators --- see Section~\ref{sec:iterator-invalidation}).
    \item Worst case $O(n)$ (pathological hashing / attacks); ordered trees are more predictable.
\end{itemize}

\begin{lstlisting}[language=C++]
std::unordered_map<std::string, int> hist;
++hist["buy"];
hist.reserve(1024);   // buckets; can avoid rehash during fill
\end{lstlisting}

Default underlying containers: \texttt{deque} for \texttt{stack}/\texttt{queue}, 
\texttt{vector} for \texttt{priority\char`_queue}. They expose a narrow interface 
(\texttt{push}/\texttt{pop}/\texttt{top} or \texttt{front}), not full random access.

\begin{lstlisting}[language=C++]
std::priority_queue<int> pq;
pq.push(3); pq.push(10); pq.push(1);
int best = pq.top();   // 10
pq.pop();
\end{lstlisting}

\subsection{Views}
\label{subsec:container-views}
See~\cite{cppreference-span},~\cite{cppreference-string-view}.
\newline
Views are \textbf{non-owning}: they refer to existing contiguous data (or a string) 
without allocating. Prefer them for function parameters that should not force a copy.
(Also covered in Contemporary C++: \texttt{std::string\char`_view}, \texttt{std::span}.)

\begin{table}[H]
    \begin{tabularx}{\textwidth}{|l|X|}
    \hline
    \href{https://en.cppreference.com/w/cpp/container/span}{span} & non-owning view over a contiguous sequence (C++20) \\
    \hline
    \href{https://en.cppreference.com/w/cpp/string/basic_string_view}{string\char`_view} & non-owning view over a character sequence (C++17) \\
    \hline
    \end{tabularx}
\end{table}

\begin{lstlisting}[language=C++]
void sum(std::span<const int> s);     // works for array, vector, C array, ...

std::vector<int> v{1, 2, 3, 4};
sum(v);
sum(std::span{v}.subspan(1, 2));      // view into the middle

void print(std::string_view sv);      // no std::string allocation at the boundary
print("literal");
print(std::string{"temp"});           // OK while the temporary lives for the full call
\end{lstlisting}

\textbf{Caution:} do not return a \texttt{string\char`_view}/\texttt{span} that refers to a 
temporary or locals that are about to die.

\section{Invalidation of Iterators and References}
\label{sec:iterator-invalidation}
Rules of thumb:
\begin{itemize}
    \item \texttt{vector}: reallocation invalidates \textbf{everything}; without reallocation, 
          insert/erase invalidates at/after the modified point.
    \item \texttt{deque}: inserting/erasing at ends keeps references to other elements 
          (iterators may still be invalidated --- treat carefully); modifying the middle 
          is more destructive.
    \item \texttt{list} / \texttt{forward\char`_list}: insert/erase does not invalidate other iterators/references.
    \item Associative (ordered): only erased elements are invalidated.
    \item Unordered: \textbf{rehash} invalidates iterators (references to elements typically remain); 
          without rehash, only erased iterators are invalidated.
\end{itemize}
